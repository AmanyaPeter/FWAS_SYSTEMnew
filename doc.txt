For FWAS, the architecture defines how the web interface, the business logic layer, and the SQLite database are organized and how data flows between them.Architectural DesignArchitectural design is the process of making and documenting those high-level structural decisions before detailed coding begins. It translates requirements into a blueprint that development teams can implement consistently. For FWAS, this meant deciding on a layered architecture, selecting PHP as the backend language, and defining how each layer communicates.Why Architecture Bridges Requirements and DesignRequirements describe what the system must do — for example, 'a finance officer must be able to register a supplier and record a payment.' Architecture answers how those requirements will be realized structurally, without yet specifying code-level implementation detail. It is the bridge because it forces abstract needs to become concrete structural choices: a requirement for offline operation became the decision to use a local Apache/PHP/SQLite stack; a requirement for role-based access became a dedicated authentication layer.Activity 2.2 — Architecture in Practice (FWAS as University Finance System)Treating FWAS as the focus system, the five major components and their interactions are as follows:ComponentDescription and InteractionsPresentation LayerHTML/CSS/JS views rendered by PHP. Its primary function is to capture, validate, and record discrete financial transactions (supplier registrations, payment entries, batch assignments) and produce outputs (batch summaries, exported spreadsheets) that feed into an external transaction system (IFMS). Every key interaction in the system — recording a payment, adding it to a batch, finalizing and exporting — is a transaction with defined pre-conditions, validation rules, and state changes. The payment status lifecycle (Draft → Ready → Batched → Processed) is characteristic of TPS design, where each record moves through defined states as it is processed.Activity 6.2 — Real-World Analysis: FWAS at MUST Finance OfficeFieldDetailOrganisationMbarara University of Science and Technology (MUST) — Finance DepartmentSystem TypeTransaction Processing System (payment preparation and batching)Architecture Applied3-tier Layered Architecture with MVC; local deployment on Apache/PHP/SQLiteChallenges ObservedFinance officers must re-enter supplier TIN numbers for every new payment transaction in IFMS. Most Challenging ConceptThe most challenging concept in Week 5 was correctly applying the 4+1 architectural views without conflating them. Initially, the group confused the Process View (runtime behavior) with the Development View (code structure). The distinction became clear when we asked: 'Is this describing what the code looks like on disk, or what happens when it runs?' For FWAS, the physical view was also non-trivial — we had to be precise that the deployment is genuinely local with no cloud component, which is architecturally significant given the offline requirement.2. Architectural Decision the Group Struggled WithThe group debated whether to use a full MVC framework like Laravel versus a plain PHP MVC structure. Laravel would have provided scaffolding, routing, and ORM features, but it introduces Composer dependency management and requires internet access during setup — incompatible with the offline deployment requirement discovered during Week 3 modeling. We resolved this by choosing plain PHP MVC, accepting more manual code in exchange for a stack that installs and runs entirely offline from a USB drive if necessary.3. How We Would Improve the DesignIf extending FWAS to version 2, three architectural improvements would be prioritized. First, a formal approval workflow for the Finance Supervisor (UC-16) would require adding an Approval state to the Batch status enum and an ApprovalController — the current architecture supports this without restructuring. Second, replacing the manual IFMS export-and-retype workflow with a direct API integration (if IFMS exposes one) would eliminate the last manual step entirely, though this depends on government infrastructure availability. Third, adding automated database backup scheduling would remove the dependency on the Admin manually triggering UC-22 — this could be implemented as a PHP cron job writing timestamped SQLite copies to a backup folder.Bonus: Architecture ComparisonLayered Architecture vs Client-ServerLayered ArchitectureClient-ServerAdvantagesClear separation of concerns; each layer independently maintainable; changes to one layer minimally affect othersNatural fit for networked systems; server centralizes data and logic; clients can be thin and replaceableDisadvantagesCan introduce overhead if too many layers; lower layers cannot access higher ones (strict call direction)Requires network infrastructure; server becomes single point of failure; latency sensitiveBest Use CaseBusiness applications with clear logic separation (ERP, financial systems, transaction processors)Distributed systems needing centralized data access (email, web services, banking backends)MVC vs Repository PatternMVCRepository PatternAdvantagesWidely understood; clear separation of UI, logic, and data; good for CRUD-heavy web applicationsAbstracts data source completely; controllers never write SQL; easy to swap databases without touching business logicDisadvantagesModel can become bloated if it handles both business rules and data access; thin models shift logic to controllersAdds abstraction overhead for simple systems; overkill if the data source will never changeBest Use CaseWeb applications with a clear UI/logic/data split — particularly form-driven transaction systems like FWASEnterprise applications where the data source may change (e.g., migrating from SQL to NoSQL, or supporting multiple data providers)FWAS — Group 9 — Software Engineering Week 5 — MUST 2026
